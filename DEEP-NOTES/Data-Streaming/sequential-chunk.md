# Sequential Chunk

## Overview

This note covers **sequential chunk streaming**: transmitting data chunks one at a time in strict order over a single stream or connection. Each chunk waits for the previous to complete before sending. Order is preserved naturally without extra logic. The trade-off is head-of-line blocking, one delayed chunk stalls everything behind it. **Goal:** recognize when order matters more than raw throughput, and implement without accidental complexity.

## Definition

**Sequential chunk streaming** means dividing data into chunks and transmitting them one after another over the same stream or connection. The sender waits for acknowledgment (or completion) of chunk N before sending chunk N+1. The receiver processes chunks immediately as they arrive in the exact order they were sent.

Key properties:

- Single stream or connection carries all chunks
- Natural ordering, no sequence numbers needed for reassembly
- Simple state management at both sender and receiver
- Head-of-line blocking, one lost or slow chunk delays all subsequent chunks

## The Analogy

**Sequential streaming is like a single-lane drive-through.** Cars (chunks) arrive one by one in exact order. The kitchen processes each order as the car pulls up. No one skips ahead. If one car stalls or takes too long ordering, everyone behind waits. Simple to manage, predictable flow, but you cannot bypass a blockage.

## When You See It

Sequential streaming appears when:

- You want simplicity and minimal memory overhead at the receiver
- The consumer processes data as it arrives without buffering for reordering
- Order is critical and must be preserved (video frames, audio samples, LLM tokens)
- The protocol naturally enforces ordering (HTTP/1.1, WebSocket messages, Server-Sent Events)

Common scenarios:

- **Video streaming**: frames must decode in order, B-frames depend on previous I/P-frames
- **Audio streaming**: samples played in sequence, out-of-order causes glitches
- **LLM token generation**: tokens stream to user in generation order for readable output
- **Real-time dashboards**: events displayed as they arrive, order implies causality
- **Command streams**: instructions must execute in the sequence sent

## Examples

**Good: Video frame streaming with natural ordering**

```typescript
/**
 * Video frame with sequence metadata.
 * @description Frame data for sequential streaming.
 */
interface VideoFrame {
  /** Frame sequence number */
  sequence: number
  /** Frame binary data */
  data: Uint8Array
  /** Frame timestamp in milliseconds */
  timestamp: number
}

/**
 * Stream video frames sequentially.
 * @description Enforces ordered frame delivery with ACK per frame.
 */
class VideoStreamer {
  /** Video encoder instance */
  private encoder: VideoEncoder
  /** Stream controller for frames */
  private streamController: ReadableStreamController<VideoFrame>

  /**
   * Initialize streamer with encoder.
   * @description Sets up video encoder instance.
   * @param encoder - Video encoder instance
   */
  constructor(encoder: VideoEncoder) {
    this.encoder = encoder
  }

  /**
   * Enqueue frame and await acknowledgment.
   * @description Blocks until frame is sent and ACK received.
   * @param frame - Video frame to stream
   */
  async streamFrame(frame: VideoFrame): Promise<void> {
    await this.sendChunk(frame)
    await this.waitForAck(frame.sequence)
  }

  /**
   * Transmit frame over single stream.
   * @description Sends frame via POST with sequence header.
   * @param frame - Video frame to transmit
   */
  private async sendChunk(frame: VideoFrame): Promise<void> {
    await fetch('/stream', {
      method: 'POST',
      body: frame.data,
      headers: {
        'X-Sequence': String(frame.sequence),
        'Content-Type': 'video/frame'
      }
    })
  }

  /**
   * Block until server ACK received.
   * @description Lost ACK stalls entire pipeline.
   * @param sequence - Frame sequence number
   */
  private async waitForAck(sequence: number): Promise<void> {
    return new Promise(resolve => {
      this.onAck(sequence, resolve)
    })
  }
}
```

**Good: LLM token streaming for real-time display**

```typescript
/**
 * LLM token with metadata.
 * @description Token generated by language model.
 */
interface Token {
  /** Token identifier */
  id: number
  /** Token text content */
  text: string
  /** Token log probability */
  logprob: number
}

/**
 * Stream LLM tokens sequentially.
 * @description Preserves generation order for readable output.
 */
class TokenStreamer {
  /** Controller for token stream */
  private controller: ReadableStreamDefaultController<Token>

  /**
   * Create readable stream for tokens.
   * @description Returns stream that enqueues tokens sequentially.
   * @param prompt - Input prompt for generation
   * @returns Readable stream of tokens
   */
  async streamTokens(prompt: string): Promise<ReadableStream<Token>> {
    return new ReadableStream({
      start: controller => {
        this.controller = controller
        this.generateAndStream(prompt)
      }
    })
  }

  /**
   * Generate and stream tokens sequentially.
   * @description Enqueues tokens one at a time.
   * @param prompt - Input prompt for generation
   */
  private async generateAndStream(prompt: string): Promise<void> {
    const tokens = this.generateTokens(prompt)
    for (const token of tokens) {
      this.controller.enqueue(token)
      await this.flushToClient()
    }
    this.controller.close()
  }

  /** Yield to event loop for flush */
  private async flushToClient(): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, 0))
  }
}
```

**Bad: Sequential for independent chunks**

```typescript
/**
 * Sequential upload for independent chunks.
 * @description Head-of-line blocking limits throughput.
 */
class SlowFileUploader {
  /**
   * Upload chunks one at a time.
   * @description Sequential head-of-line blocking.
   * @param file - File to upload
   */
  async uploadLargeFile(file: File): Promise<void> {
    const chunks = this.splitIntoChunks(file, 1024 * 1024)
    for (const chunk of chunks) {
      await fetch('/upload', {
        method: 'POST',
        body: chunk
      })
    }
  }
}

/**
 * Better: Parallel upload for independent chunks.
 * @description Controlled concurrency with multiple chunks in flight.
 */
class FastFileUploader {
  /**
   * Split file and upload with parallelism.
   * @description Batches chunks for concurrent upload.
   * @param file - File to upload
   */
  async uploadLargeFile(file: File): Promise<void> {
    const chunks = this.splitIntoChunks(file, 1024 * 1024)
    const concurrency = 4
    for (let i = 0; i < chunks.length; i += concurrency) {
      const batch = chunks.slice(i, i + concurrency)
      await Promise.all(batch.map(chunk => fetch('/upload', { method: 'POST', body: chunk })))
    }
  }
}
```

## Important Points

- **Natural ordering** means no sequence numbers or reassembly buffer needed at the receiver
- **Head-of-line blocking** is the main drawback, one slow chunk stalls the entire pipeline
- **Memory efficiency** at receiver, chunks processed immediately without buffering for reorder
- **Simple state** makes debugging and reasoning easier
- **HTTP/1.1, WebSockets, SSE** naturally support sequential streaming
- **Chunk size** affects latency, smaller chunks mean more frequent round trips
- **Not suitable for** bulk transfers where order does not matter and throughput is priority

## Summary

- **Sequential**: chunks flow one-by-one over single stream, natural order, simple implementation
- **Trade-off**: simplicity and ordering vs throughput and head-of-line blocking
- **Analogy**: single drive-through lane, cars processed in arrival order
- **Use for**: ordered streams (video, audio, LLM tokens), real-time processing, causal event streams
- **Avoid when**: chunks are independent and throughput matters more than ordering

_Sequential streaming is the right choice when the story must be told in order. Accept the head-of-line blocking as the cost of coherence._
